nemo_retriever.service.routers.admin module#

Admin endpoints exposed by worker pods (realtime / batch / standalone).

These are NOT part of the public ingest contract — they exist so the gateway’s dashboard overview can fan out and ask each worker pod “what’s your live queue depth?” without going through Prometheus. Available without auth so the cluster’s internal Service DNS can reach them; the routes are not advertised on the OpenAPI schema.

The same data is published as Prometheus metrics (nemo_retriever_pool_queue_depth*()) — this endpoint is the human-readable, JSON-shaped sibling for use cases that can’t or don’t want to go through Prometheus.

async nemo_retriever.service.routers.admin.pool_stats(

) → dict[str, Any][source]#

Return live worker-pool state for the current pod.

Response shape:

{
  "mode": "realtime" | "batch" | "standalone" | "gateway",
  "pools": {
    "realtime": {
      "queue_depth": int,
      "queue_depth_ratio": float,
      "max_queue_size": int,
      "num_workers": int,
      "processed": int,
      "is_running": bool,
    },
    ...
  }
}

Gateway pods have no local pool and return {"mode": "gateway", "pools": {}} — the caller is expected to fan out to the worker Services themselves for live numbers.

SSE stream of job-tracker events with periodic summary heartbeats.

Subscribes to the global event bus (no job_id filter) so the dashboard sees both per-document events and the J5 job lifecycle events for every job. nemo_retriever.service.routers.ingest module#

Ingest endpoints: general, per-page, and whole-document upload.

Each endpoint is mode-aware:

  • gateway — record IngestMetrics (the single authoritative store), then proxy the raw HTTP request to the correct backend worker pod.

  • standalone — record IngestMetrics and enqueue locally.

  • realtime / batch — enqueue work to the local pipeline pool only. IngestMetrics is not initialised in worker modes, so get_metrics() returns None and no per-item tracking occurs.

async nemo_retriever.service.routers.ingest.create_job(
request: Request,
body: JobCreateRequest,
) → JobCreatedResponse[source]#

Open a job that will receive expected_documents uploads.

The server returns an opaque job_id the client uses for every subsequent POST /v1/ingest/job/{job_id}/document (or /page) call. The job is in-memory only; gateway pod restarts erase it (this is intentional — see the J1 design notes).

async nemo_retriever.service.routers.ingest.get_job_document(
request: Request,
job_id: str,
document_id: str,
) → Response[source]#

Single-document detail nested under the owning job.

Returns HTTP 200 when the document is terminal (consumes result_data so memory is freed for the caller) and HTTP 202 while still pending/processing. A 404 is returned if either the job is unknown or the document does not belong to this job — the latter prevents leaking document existence across tenants.

Stream Server-Sent Events for a single JobAggregate.

Emits:

  • a catch-up batch of already-terminal completed / failed document events (so clients that connect after some documents already finished still see them),

  • live per-document processing / completed / failed events,

  • lifecycle events (job_progress, job_finalized, job_failed) — these arrive once J5 wires them through the tracker; the endpoint already routes them.

A keepalive comment is emitted every 30 seconds.

async nemo_retriever.service.routers.ingest.job_callback(
request: Request,
) → JSONResponse[source]#

Accept a completion notification from a worker pod.

The gateway’s JobTracker is updated and an SSE event is published so connected clients are notified instantly.

The log line emitted here is the primary diagnostic signal for “client hang” reports: it carries the job_id looked up from the tracker, the actual transition outcome (transitioned / idempotent / unknown_document), and the per-job subscriber count so operators can correlate worker-pod completion with client-side SSE delivery without grepping multiple files.

async nemo_retriever.service.routers.ingest.pipeline_config(request: Request)[source]#

Return redacted pipeline configuration.

  • worker / standalone — returns the local pipeline configs directly.

  • gateway — fans out GET requests to one realtime and one batch backend pod, aggregates the responses, and returns them keyed by role.
